# /analyze-issue 命令

分析 Issue 或需求描述，输出结构化的需求分析报告。

## 使用方式

```text
/analyze-issue [Issue 链接或需求描述]
```

## 执行流程

1. **解析需求**
   - 理解功能描述和用户期望
   - 识别显式需求和隐含需求

2. **检索上下文**
   - 查看相关现有代码（使用 Grep/Glob）
   - 检查是否有类似功能实现
   - 阅读相关设计文档

3. **评估复杂度**
   - 低（< 50 行代码）：简单 bug 修复、配置调整
   - 中（50-200 行）：新功能、模块增强
   - 高（> 200 行）：架构变更、新系统

4. **输出分析报告**

## 输出格式

```markdown
## 需求分析报告

### 1. 需求概述
[一句话描述核心需求]

### 2. 功能边界
**包含**：
- [功能点 1]
- [功能点 2]

**不包含**：
- [明确排除的内容]

### 3. 验收标准
- [ ] [可验证的标准 1]
- [ ] [可验证的标准 2]
- [ ] [可验证的标准 3]

### 4. 复杂度评估
- **级别**：低 / 中 / 高
- **预估代码量**：约 XX 行
- **涉及模块**：[模块列表]

### 5. 技术要点
- [关键技术决策 1]
- [关键技术决策 2]

### 6. 风险识别
- [潜在风险 1]
- [潜在风险 2]

### 7. 下一步建议
- [ ] 是否需要设计文档：是 / 否（简化版）
- [ ] 是否需要 ADR：是 / 否
- [ ] 建议的分支名：`feat/xxx` 或 `fix/xxx`
```
